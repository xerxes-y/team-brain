The gap it closes

The tax isn't "we don't have enough docs" — it's translation loss at the PO ↔ developer ↔ tester seam, repeated per ticket, with no durable record. A PO and a dev spend 20 minutes on why a rule exists; that reasoning lives in two heads and an unsearchable scrollback. A different dev breaks it three sprints later, or re-asks. The tester chases both people to reconcile expected vs actual.

Jira and Confluence hold the artifacts — they don't hold the reasoning that connects them. When someone rotates off the team, the why leaves with them; the ticket and the diff survive, the reasoning doesn't. team-brain pulls that reasoning into one shared, role-aware memory and serves it back the moment someone needs it, instead of an interruption.

📋 Product Owner↔ 💻 Developer↔ 🧪 Tester

Honest caveat: this only pays off if capture actually happens. It's not a search layer over docs you already have — it's a bet that automatic capture beats manual documentation, specifically at this handoff. No capture, no value.

Three roles, three bridges

Each role gets one MCP tool tuned to how it actually asks questions. All three share the same store, so an answer for one role is built from what the others already captured.

Product Owner

team_draft_ticket

"I need a ticket to finalize type-2 cash transfers." It mines the rules straight from the codebase and GitLab — so a PO who doesn't read Java still writes an accurate ticket.

Developer

team_explain_ticket

Hand it a Jira key. It returns the business intent and the code paths that implement it, bridging "business language ↔ code" both ways.

Tester

team_test_plan

The both-sides view: business rules + implementation gotchas for a ticket, turned into what actually needs testing.

Capture the work, not extra work

The brain fills two ways — passively as the team works, and deliberately when something matters.

Passive connectors

Point them once; they keep the brain current.

  • Jira — JQL paging, ADF flatten, issue-security ACL, comments
  • Confluence — CQL paging, page-restriction ACL, heading chunks, incremental
  • GitHub PRs — the "why" behind merged PRs, private-repo ACL
  • GitLab — mines business rules out of a codebase for the PO
  • Devin & IntelliJ — IDE activity: sessions, git commits, TODO/FIXME, live ACP tap

Deliberate capture

When a PO finishes analysing a ticket, or a dev finishes implementing one, push the decisions and gotchas in one move — tagged to the ticket so the role bridges pick them up instantly.

# inside the IDE chat — one of:
/team-capture ticket=PROSET-9913 role=developer
# …or just say:
"start saving team-brain"

Both nudge the agent to call team_capture; it extracts the Jira key, distills the chat into facts, and stores them as searchable vectors.

Ask it anything — get a cited answer

team_assist(query, role) retrieves by meaning, not keywords, so a tester asking "what's the riskiest edge case?" finds the developer's note about matching the counterparty account — zero shared words.

Architecture — read path

The classic split, done right: vectors find the relevant text, the LLM reads it. A raw embedding is a list of floats no model can interpret — its only job is narrowing thousands of memories to the handful that matter.

question→ embed→ pgvector + BM25 (RRF)→ ACL filter→ soft-rerank
(role bias)→ LLM reads
+ cites

Built on memento's MemoryStorePG: Postgres with BM25 tsvector + pgvector fused by reciprocal-rank fusion, namespaces, an entity graph, and an audit log. No Postgres? It runs on local SQLite for tests and solo trials.

The role bridges — different output, same retrieval

Each role gets the same search underneath, but the retrieval bias and output shape are different. One shared namespace means a PO's business rule and a dev's code gotcha live together, searchable, without duplicating the store.

PO: draft_ticket

Input: a need. Bias: business rules, decisions, specs. Output: title + background + acceptance criteria, so the PO writes an accurate ticket.

Dev: explain_ticket

Input: a Jira key. Bias: PRs, decisions, code rules. Output: business intent + code paths, bridging "why did we build this" ↔ "where is it in code."

Tester: test_plan

Input: a feature/ticket. Bias: both PO rules + dev implementation. Output: expected vs actual behavior + edge cases to test.

Access control that fails closed

Knowledge is scoped by acl:<group> tags. A memory with no ACL is public within its namespace; anything scoped is hidden unless the asker is in the group — and the default on doubt is hide, not leak. Source connectors carry their origin's restrictions through (Jira issue-security, Confluence page restrictions, private-repo access), so the brain never exposes what its sources wouldn't.

Embeddings: laptop to GPU, one env var

The embedder is an adaptor — pick a resource profile, team-brain chooses backend, model, and dimension. Start on a laptop with no API key; move to a GPU box later with a single change, then re-index.

Why this shape

Hybrid search (BM25 + pgvector)

Keywords find the exact phrase; semantics find the meaning. Fusing both catches "riskiest edge case" even if no tester used those exact words.

ACL as tags (fail-closed)

No schema change to Postgres. Connectors just add acl:group tags. Unknown asker → see public only. Leaked Confluence page is hard to reverse.

One namespace, roles reshape retrieval

Dev answer can pull PO's business rule. Tester sees both. Bridges the gap instead of building three silos.

Search → then synthesize

Retrieval bounds the LLM. Wrong search → confident wrong answer. Extractive fallback (no LLM) still works.

Install in 2 minutes

It's an MCP server. Install it, point your IDE agent at it, done — the storage engine is pulled in automatically.

# 1) install (memento engine comes with it)
pip install git+https://github.com/xerxes-y/team-brain.git

# 2) register in your MCP client's config
{
  "mcpServers": {
    "team-brain": {
      "command": "team-brain",
      "env": { "MEMENTO_DB_URL": "postgresql://…", "TEAMBRAIN_EMBED": "local" }
    }
  }
}

Solo trial needs nothing else — drop MEMENTO_DB_URL and it uses local SQLite. For a shared team brain, point every member at the same Postgres. Full Devin wiring: docs/devin-mcp.md.
